# 🏠 Automation Test

본 프로젝트는 **Playwright**를 활용하여 **호갱노노(HoGangNoNo)** 서비스의 핵심 사용자 기능을 자동화한 테스트 프로젝트입니다. 자동화 구조는 테스트 유지보수성과 확장성을 높이기 위해 **Page Object Model(POM)** 기반으로 설계되었으며, 반복적인 시나리오를 안정적으로 검증할 수 있도록 구성되었습니다.

---

## 📌 프로젝트 개요

본 프로젝트는 실제 사용자가 서비스를 이용하는 핵심 플로우 중심으로 자동화를 구축했습니다.

* 로그인 이후 주요 기능 사용 흐름 자동화
* 지도 기반 아파트 검색 기능 검증
* 아파트 상세 페이지 정보 렌더링 확인
* 즐겨찾기 등록/삭제 기능 검증
* 핵심 사용자 경험(UX) 흐름을 우선적으로 자동화

테스트는 사용자의 실제 행동을 기준으로 설계되었으며, 서비스 변경에 유연하게 대응할 수 있도록 구조화했습니다.

---

## 📁 프로젝트 구조

```
e2e/
 ├─ pages/                # 페이지 객체 모음
 │   ├─ login.page.ts     # 로그인 페이지
 │   ├─ map.page.ts       # 지도/검색 페이지
 │   ├─ detail.page.ts    # 아파트 상세 페이지
 │   └─ base.page.ts      # 공통 페이지 로직
 │
 ├─ tests/                # 테스트 시나리오
 │   ├─ login.spec.ts
 │   ├─ search.spec.ts
 │   └─ favorite.spec.ts
 │
 ├─ fixtures/             # 공통 테스트 설정
 │
 └─ playwright.config.ts
```

## 🔎 자동화한 기능 목록

### 1) 로그인 기능

* 정상 로그인 흐름 검증
* 잘못된 인증 정보 입력 시 오류 메시지 확인

### 2) 지역/아파트 검색 기능

* 검색창을 통한 지역 및 아파트명 검색
* 지도 렌더링 및 검색 결과 확인

### 3) 상세 페이지 기능

* 특정 아파트 선택 후 상세 정보 접근
* 실거래가, 호가, 단지 정보 등 렌더링 검증

### 4) 즐겨찾기 기능

* 즐겨찾기 추가 및 삭제 동작 확인
* 즐겨찾기 목록의 정상 반영 여부 확인

## 🧱 유지보수 전략

본 자동화 프로젝트는 장기적으로 안정적인 테스트 운영을 위해 아래 전략을 기반으로 구조화되었습니다.

### ✔️ 페이지 변경 시 최소 수정

* UI 변경이 발생해도 Page 파일만 수정하면 전체 테스트가 정상 동작하도록 설계

### ✔️ 사용자 행동 중심의 테스트 시나리오

* 테스트 코드는 "무엇을 검증하는가"에 집중
* 세부 동작은 Page 객체 내부에 분리해 캡슐화

### ✔️ 서비스 구조 특성을 반영한 모듈 구성

* 지도 기반 UI와 같은 복잡한 기능은 별도 페이지 객체로 구분
* 반복 동작(검색, 페이지 이동 등)은 공통 메서드화

### ✔️ 테스트 독립성 보장

* 테스트 간 상호 영향이 없도록 초기 상태 정리 후 수행
* 즐겨찾기, 검색 상태 등은 각 테스트별로 리셋

### ✔️ CI/CD 통합 고려

* 병렬 실행을 고려한 테스트 구성
* 환경별 설정(dev/stage/prod) 분리 가능 구조

## 📚 결론

이 자동화 테스트 프로젝트는 Playwright 기반의 구조적 자동화를 통해 호갱노노 서비스의 핵심 사용자 흐름을 안정적으로 검증하는 것을 목표로 합니다.

* 사용자 중심 시나리오
* 유지보수 친화적인 구조
* 서비스 변경에 유연하게 대응 가능한 설계

을 기반으로 실제 서비스의 품질 안정성을 향상시키는 데 집중했습니다.

포트폴리오 용도로 작성되었으며, 실무에서도 확장 가능한 자동화 프레임워크 구조를 반영하고 있습니다.
