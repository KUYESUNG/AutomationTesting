# 🏠 Automation Test

본 프로젝트는 **Playwright**를 활용하여 **호갱노노(HoGangNoNo)** 서비스의 핵심 사용자 기능을 자동화한 테스트 프로젝트입니다. 자동화 구조는 테스트 유지보수성과 확장성을 높이기 위해 **Page Object Model(POM)** 기반으로 설계되었으며, 반복적인 시나리오를 안정적으로 검증할 수 있도록 구성되었습니다.

---

## 📌 프로젝트 개요

본 프로젝트는 실제 사용자가 서비스를 이용하는 핵심 플로우 중심으로 자동화를 구축했습니다.

* 로그인 이후 주요 기능 사용 흐름 자동화
* 단지 검색 기능 검증
* 알림 등록/삭제 기능 검증
* 핵심 사용자 경험(UX) 흐름을 우선적으로 자동화

테스트는 사용자의 실제 행동을 기준으로 설계되었으며, 서비스 변경에 유연하게 대응할 수 있도록 구조화했습니다.

---

## 📁 프로젝트 구조

```
e2e/
 ├─ pages/                    # 페이지 객체 모음
 │    ├─ login                # 로그인 
 │    │   └─ login.page.ts
 │    ├─ search               # 단지 검색 
 │    │   └─ search.page.ts
 │    └─ notification         # 알림 설정
 │        └─ notification.page.ts
 ├─ tests/                # 테스트 시나리오
 │    ├─ login                
 │    │   └─ login.spec.ts
 │    ├─ search               
 │    │   └─ search.spec.ts
 │    └─ notification
 │        └─ notification.page.ts
 │
 └─ utils/             # 공통 테스트 설정
```

## 🔎 자동화한 기능 목록

### 1) 로그인 기능

* 정상 로그인 흐름 검증
* 잘못된 인증 정보 입력 시 오류 메시지 확인

### 2) 단지 검색 기능

* 검색창을 통한 단지명 검색
* 지도 렌더링 및 검색 결과 확인

### 3) 알림 설정 기능

* 알림 추가 및 삭제 동작 확인
* 관심 목록의 정상 반영 여부 확인

## 📌 Playwright를 채택 이유

* 테스트 실행 시 요소 로딩 여부를 자동으로 판단해 대기해주기 때문에, 불필요한 wait 코드 작성 없이도 안정적으로 테스트를 구성할 수 있음

* 실제 브라우저 환경과 매우 유사하게 동작하여 사용자 흐름을 자연스럽게 재현하기 쉬움

* 여러 브라우저 환경을 하나의 구조로 다룰 수 있어 초기 설정과 운영이 간단함

* 코드 생성 기능을 지원해 테스트 초안을 빠르게 만들 수 있어 자동화 초기에 속도감을 가지고 구축 가능

---

## 📌 TypeScript 선택 이유

* 타입 기반 언어라 자동화 코드 작성 시 실수를 미리 잡아주고, 런타임 오류 가능성을 줄여 안정적인 테스트 환경을 구성할 수 있음

* IDE 자동완성 지원이 강력해 테스트 코드 작성 속도와 정확도가 높아짐

* 구조화된 코드 작성이 쉬워 테스트 규모가 커져도 유지보수하기 수월함

* 테스트 로직을 명확하게 표현할 수 있어 시나리오 단위 관리가 깔끔함

---

## 📌 POM(Page Object Model) 구조 채택 이유

* 페이지 단위로 요소와 동작을 분리해두면 UI 수정이 발생해도 테스트 전체가 아닌 해당 페이지 파일만 수정하면 되어 유지보수 부담이 크게 줄어듦

* 테스트 코드는 검증 목적만 남기고 실제 동작은 페이지 객체에서 처리하도록 구성해 가독성이 뛰어나고 시나리오 중심 접근이 쉬움

* 반복해서 사용하는 행동(로그인, 검색 등)을 한 번만 정의해 재사용할 수 있어 테스트 설계 효율이 높아짐

* 기능이 계속 추가되어도 페이지 파일을 확장하는 방식으로 자연스럽게 구조가 커질 수 있어 장기적인 테스트 운영에 적합함
